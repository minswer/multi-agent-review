# Multi-Agent 코드 리뷰 워크플로우

이 저장소의 PR 변경분은 **오케스트레이터(메인 Claude) + 3개의 서브에이전트**가 병렬로 리뷰한다.

## 역할

- **오케스트레이터(메인 Claude)**: 변경 파일 수집, 서브에이전트 병렬 호출, 결과 취합, GitHub PR 코멘트 게시.
- **security-reviewer** (`sonnet`, `isolation: worktree`): 보안 취약점 (SQL Injection, XSS, 시크릿 노출, 명령어 인젝션).
- **performance-reviewer** (`sonnet`, `isolation: worktree`): N+1 쿼리, 동기 I/O, 메모리 낭비, 캐싱 기회.
- **style-reviewer** (`haiku`, `isolation: worktree`): 명명 규칙, 함수 길이(50줄+), 매직 넘버, 중복 코드.

> 서브에이전트는 서로를 직접 호출하지 않는다. 항상 오케스트레이터를 경유한다.

## 트리거

- 로컬: 사용자가 "병렬 리뷰해줘"라고 요청
- CI: `pull_request` 이벤트 (`opened`, `synchronize`) — `.github/workflows/claude-review.yml`

## 표준 절차

1. **변경 파일 수집**
   ```bash
   git diff main...HEAD --name-only
   ```
   3-dot diff로 PR 분기 시점 이후의 변경만 추출.

2. **PR 크기 가드**
   변경 파일이 10개 초과면 분석을 생략하고 PR에 다음 코멘트를 작성:
   > 변경량이 너무 많습니다. 10개 이하로 분리하는 것을 권장합니다.

3. **병렬 호출** — Task 도구로 세 서브에이전트를 동시에 실행. 각자 `isolation: worktree`로 독립 Git Worktree에서 동작.

4. **결과 파일 저장 규칙 (각 서브에이전트의 의무 출력)**

   각 에이전트는 자신의 분석 결과를 정확히 아래 경로에 저장한다. 다른 경로로 저장하지 않는다.
   필요 시 디렉터리는 `mkdir -p /tmp/review` 로 먼저 생성한다.

   | 에이전트 | 저장 경로 |
   |---|---|
   | `security-reviewer` | `/tmp/review/security.md` |
   | `performance-reviewer` | `/tmp/review/performance.md` |
   | `style-reviewer` | `/tmp/review/style.md` |
   | **오케스트레이터(최종)** | `/tmp/review/final.md` |

   오케스트레이터는 위 세 파일을 모두 읽은 뒤 `/tmp/review/final.md`를 작성한다.
   세 파일 중 하나라도 비어 있거나 누락되면 해당 에이전트를 재호출한다.

5. **취합 → 최종 리뷰 (`/tmp/review/final.md` 작성 기준)**

   세 에이전트의 결과를 통합할 때 반드시 다음 3가지 기준을 적용한다.

   **5-1. 중복 제거**
   - 동일한 `파일:라인`을 지적하는 이슈가 여러 리뷰어에서 나오면, **가장 심각도가 높은 것 하나만** 유지한다.
   - 동률이면 보안 > 성능 > 스타일 순.
   - 제거된 이슈는 남긴 이슈의 "관련 지적" 항목에 한 줄 참조만 남긴다.

   **5-2. 우선순위 매트릭스 (필수 섹션)**
   - 모든 이슈를 `(수정 난이도) × (영향도)` 의 3×3 격자에 배치한다.
   - 난이도: **쉬움 / 중간 / 어려움**
   - 영향도: **높음 / 중간 / 낮음**
   - 매트릭스는 마크다운 표로 표현하고, 각 칸에 해당 이슈의 짧은 식별자(`[보안][높음] users.py:45` 형식)를 나열한다.
   - **읽는 순서: "쉬움 × 높음" → "중간 × 높음" → "쉬움 × 중간"** 순으로 정렬해 보여준다 (ROI 높은 순).

   **5-3. 높은 심각도 요약 섹션 (필수)**
   - `final.md` 최상단에 `## 🔴 즉시 수정 필요 (높은 심각도)` 섹션을 둔다.
   - 여기에는 **`[높음]` 등급 이슈만** 추출해 한 줄 요약 + 파일:라인 + 1줄 권장으로 정리.
   - 본문 상세 섹션이 길어도 리뷰어가 이 섹션만 봐도 다음 액션을 알 수 있게 한다.

   **최종 final.md 구조 (이 순서 고정)**
   ```
   ## 🔴 즉시 수정 필요 (높은 심각도)   ← 5-3 요약
   ## 📊 우선순위 매트릭스             ← 5-2 매트릭스
   ## 상세 결과                       ← 심각도 순(높음→중간→낮음), 5-1 적용 후
   ## 분석 메타                       ← 변경 파일 수, 각 리뷰어 실행 상태
   ```

6. **PR 코멘트 게시**
   `gh pr comment <PR번호> --body-file /tmp/review/final.md`

## 출력 형식 (각 서브에이전트 공통)

```
[심각도] 파일:라인 - 제목
  근거 / 권장 수정
```

## 비용 정책

- `security-reviewer`, `performance-reviewer`는 추론이 필요하므로 `sonnet`.
- `style-reviewer`는 패턴 매칭 위주이므로 `haiku` (약 30% 비용 절감).
- 모든 에이전트는 최소 권한 도구 세트만 부여.
